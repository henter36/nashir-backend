## 5. Product Data Model Planning Decision

Decision: Future product persistence requires a workspace-scoped product data model.

Minimum conceptual product record responsibilities:

- stable product identifier;
- owning workspace identifier;
- product payload fields aligned with OpenAPI authority;
- created timestamp;
- updated timestamp;
- version or revision field for optimistic concurrency;
- lifecycle state only if supported by accepted product authority;
- audit correlation support if later authorized;
- idempotency linkage for create behavior if later authorized.

This is conceptual only.

This gate does not define SQL columns.

This gate does not create a database schema.

This gate does not authorize table creation.

## 6. Workspace Ownership Decision

Decision: Product data must be workspace-owned.

Rules for future data model planning:

- every product must belong to exactly one workspace;
- product queries must include `workspaceId`;
- product ID alone must never authorize access;
- cross-workspace product access must resolve to non-disclosing 404;
- body-provided `workspaceId` remains untrusted;
- route-derived workspace context remains authoritative;
- repository boundaries must require workspace context.

This gate does not implement ownership enforcement.

## 21. Risk Review

### 21.1 Contract Drift Risk

Risk:

The product data model may introduce fields or lifecycle states not supported by OpenAPI authority.

Mitigation:

Require a Product Data Model Authority Review Gate before SQL/migration planning.

### 21.2 Workspace Isolation Risk

Risk:

A product schema that allows product lookup without workspace ownership may leak cross-workspace data.

Mitigation:

Require workspace ownership as a data model invariant.

### 21.3 Premature Migration Risk

Risk:

Planning may be mistaken as authorization to create migrations.

Mitigation:

State explicit NO-GO for SQL and migrations.

### 21.4 Test DB Risk

Risk:

Persistence implementation may start without a safe test database lifecycle.

Mitigation:

Require test DB strategy before implementation.

### 21.5 Idempotency/Concurrency/Audit Risk

Risk:

Product writes may be implemented without idempotency, optimistic concurrency, or audit persistence.

Mitigation:

Keep create/update implementation blocked until these storage concerns are planned.
