## 5. Reviewed Planning Decisions

### 5.1 AuditRepository Target

Decision:

Proceed with planning toward a dedicated `AuditRepository`.

Review outcome:

Accepted.

Reason:

A repository boundary keeps audit persistence separate from product route handling and avoids embedding raw SQL audit writes directly inside product handlers.

### 5.2 Audit Persistence Target

Decision:

Use the existing `audit_events` persistence target if the approved schema is sufficient.

Review outcome:

Accepted conditionally.

Condition:

The implementation authorization gate must verify the existing `audit_events` schema before implementation. If the schema is insufficient, a separate migration planning/authorization gate is required before runtime work.

### 5.3 Audit Event Shape

Decision:

The future audit event record shape should include the following fields:

| Field | Review Decision |
| :--- | :--- |
| `workspaceId` | Required |
| `actorId` | Required |
| `action` | Required |
| `resourceType` | Required |
| `resourceId` | Required |
| `correlationId` | Required |
| `occurredAt` | Required on persisted/returned audit event; generated by backend or database, not trusted from client input |
| `metadata` | Optional structured metadata, minimal and non-sensitive |

Review outcome:

Accepted for implementation authorization planning.

### 5.4 `occurredAt` Boundary

Decision:

`occurredAt` must not be accepted from product route request bodies.

Review outcome:

Accepted.

Implementation direction:

The implementation authorization gate should choose one of:

| Option | Description | Review Preference |
| :--- | :--- | :--- |
| Database-generated timestamp | Repository relies on DB-generated timestamp such as `created_at` | Preferred if current schema supports it |
| Backend-generated timestamp | Repository sets timestamp from trusted backend time | Acceptable if schema requires explicit value |

The implementation authorization gate must not trust client-provided timestamps for audit chronology.

### 5.5 Product Create Audit Behavior

Decision:

Successful product create should write one audit event.

Review outcome:

Accepted.

Constraints:

- First successful create writes `product.created`.
- Idempotency replay must not write a duplicate audit event.
- Failed create must not write a successful create audit event.

### 5.6 Product Update Audit Behavior

Decision:

Successful product update should write one audit event.

Review outcome:

Accepted.

Constraints:

- Successful update writes `product.updated`.
- Version conflict must not write update audit event.
- Not found must not write update audit event.
- Validation failure must not write update audit event.

### 5.7 Read/List Audit Behavior

Decision:

Do not audit product read/list endpoints in V1.

Review outcome:

Accepted.

Reason:

V1 should focus on mutation traceability and keep scope narrow.

### 5.8 Transaction Boundary

Decision:

Product mutation and required audit write should be part of one reliable consistency boundary.

Review outcome:

Accepted as a required decision for implementation authorization.

Preferred V1 direction:

Use the same database transaction for product create/update and audit event write if current repository transaction design supports it.

Fallback:

If same-transaction implementation requires broad refactoring, the implementation authorization gate must explicitly document the narrower feasible option and its risk.

### 5.9 Audit Failure Behavior

Decision:

Do not silently ignore audit persistence failure for audited product mutations.

Review outcome:

Accepted.

Preferred V1 behavior:

If required audit write fails during product create/update, the mutation should fail within the same transaction boundary.

### 5.10 Metadata Boundary

Decision:

Audit metadata must be minimal, structured, and non-sensitive.

Review outcome:

Accepted.

Allowed metadata examples:

| Operation | Metadata |
| :--- | :--- |
| `product.created` | `productId`, `status`, `idempotencyKey`, `version` |
| `product.updated` | `productId`, `previousVersion`, `newVersion`, `changedFields` |

Blocked metadata:

| Blocked Data | Reason |
| :--- | :--- |
| Raw authorization headers | Secret leakage risk |
| Full request body | Overcollection risk |
| Full product payload | Unnecessary PII/data retention risk |
| Client-provided timestamp as source of truth | Chronology trust risk |

## 8. Required Tests for Future Implementation

The future implementation authorization gate should require tests for:

| Test Area | Required Coverage |
| :--- | :--- |
| AuditRepository insert | Persists audit event with required fields |
| Product create audit | Successful create writes `product.created` |
| Product create idempotency replay | Replay does not duplicate audit event |
| Product update audit | Successful update writes `product.updated` |
| Product update conflict | Conflict does not write audit event |
| Product not found | No audit event is written |
| Validation failure | No audit event is written |
| Audit failure behavior | Required audit write failure prevents audited mutation if same transaction is selected |

## 10. Legal / Compliance Review

Audit records are compliance-sensitive.

Accepted constraints:

- Do not store secrets.
- Do not store raw tokens.
- Do not store raw request bodies.
- Do not trust client timestamps.
- Keep audit metadata minimal.
- Keep workspace and actor boundaries explicit.
- Preserve chronological audit ordering through backend or database time.
